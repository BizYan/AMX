# 资料摄取与知识入库生产闭环设计

## 目标

将项目资料上传后的同步解析过程改造成可追踪、可重试、可治理的持久化摄取闭环。上传成功只代表原文件已保存并创建摄取任务；任务完成后，资料才进入可用于文档生成、知识图谱和变更追溯的状态。

## 用户闭环

1. 项目成员上传资料，系统返回 `pending` 资料和排队中的摄取任务。
2. 用户可查看任务状态、阶段、尝试次数、错误和知识条目数量。
3. 系统或用户执行任务后，任务进入 `running`，完成后进入 `completed`；失败进入 `failed`。
4. 失败任务可重试；已完成资料可重新摄取，重新摄取前清理该来源生成的旧知识。
5. 项目负责人删除原文件时，系统同时软删除其来源知识，避免知识图谱继续使用失效证据。

## 架构

新增 `SourceIngestionJob` 持久化任务模型，承担状态、尝试次数、阶段、错误和结果证据。`SourceIngestionService` 负责任务创建、执行、重试和来源知识清理；现有 `SourceFileService.ingest_source_file` 保留为实际解析执行器。

上传和确认接口只创建任务，不在请求内执行解析。新增任务列表、执行、重试和重新摄取 API。前端项目文件页轮询资料和任务状态，向用户提供执行、重试和重新摄取操作。

## 数据与状态

任务状态：`pending | running | completed | failed | cancelled`。

每个资料同一时间最多一个活动任务。任务记录保存：

- `source_file_id`、`project_id`、`tenant_id`
- `status`、`stage`
- `attempt_count`、`max_attempts`
- `error_message`、`result_json`
- `requested_by_id`、`started_at`、`completed_at`

## 安全与治理

- 任务和资料均按租户、项目和成员权限隔离。
- 执行和重试需要项目成员权限；删除资料仍仅允许项目负责人。
- 重试和重新摄取不会生成重复活动任务。
- 删除和重新摄取会软删除来源知识及其关联关系，保留审计可追踪性。
- 错误只记录安全摘要，不记录原文件内容或凭证。

## 验证

- API 测试覆盖任务排队、执行、失败、重试、重新摄取、活动任务去重和删除治理。
- Alembic 测试验证单一迁移头。
- Web E2E 覆盖任务状态显示、执行、失败重试和重新摄取入口。
- Web 类型检查、生产构建和 GitHub CI 作为 PR 门禁。

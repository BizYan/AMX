# 受控邀请注册与项目加入闭环设计

## 1. 背景与目标

AMX 已支持项目负责人创建、续期、撤销项目邀请，也支持已登录且邮箱、租户均匹配的现有用户接受邀请。但系统没有面向新用户的受控注册激活流程，因此外部受邀人无法自行创建账号并加入项目。

本切片建立完整的受控邀请注册闭环：

1. 项目负责人向邮箱创建邀请。
2. 受邀人打开邀请链接并查看安全预检信息。
3. 已有同租户账号登录后直接接受邀请。
4. 尚无账号的受邀人设置姓名和密码，系统创建最低权限账号并加入项目。
5. 邀请消费、账号创建、角色分配和项目加入具备事务一致性、审计证据和明确失败反馈。

系统不开放自由注册。没有有效邀请令牌的访问者不能创建账号。

## 2. 范围

### 2.1 包含

- 邀请令牌公开预检接口。
- 基于有效邀请的用户激活接口。
- 新用户创建、最低权限角色分配和项目成员关系创建。
- 已有用户登录后接受邀请的现有路径兼容。
- 跨租户邮箱冲突、过期、撤销、重复接受和并发接受保护。
- 邀请激活页面及完整状态反馈。
- 审计记录、通知和测试。
- 必要的数据库约束或迁移。

### 2.2 不包含

- 无邀请的公开注册。
- 自助创建新租户。
- 邮箱所有权验证码系统或第三方身份提供商接入。
- 项目负责人自定义新用户平台级角色。
- 自动发送邀请邮件。本切片继续复用现有邀请链接交付方式。

## 3. 用户与权限模型

### 3.1 项目负责人

- 创建、查看、续期和撤销项目邀请。
- 复制邀请链接。
- 不能通过邀请授予平台管理员或高权限角色。

### 3.2 新受邀用户

- 仅能通过有效邀请令牌查看有限预检信息。
- 激活时必须使用邀请中的邮箱，不允许修改。
- 创建后获得同租户最低权限角色，并成为目标项目成员。

### 3.3 已有用户

- 同租户且邮箱匹配：登录后可以接受邀请。
- 同邮箱账号属于其他租户：拒绝激活与接受，避免跨租户身份混淆。
- 已是项目成员：返回幂等成功状态，不重复创建成员关系。

## 4. 安全边界

- 数据库只保存邀请令牌摘要，不保存新生成的明文令牌。
- 公开预检接口只返回邀请邮箱掩码、项目名称、有效期和状态，不返回租户 ID、用户列表或其他项目数据。
- 激活接口要求有效、未过期、未撤销、未接受的邀请令牌。
- 激活邮箱由邀请记录决定，客户端不能提交或覆盖邮箱。
- 密码使用现有密码策略和哈希实现。
- 激活成功后立即消费邀请，令牌不可重复使用。
- 跨租户同邮箱账号必须拒绝，不自动迁移或合并身份。
- API 对无效、过期和已撤销令牌返回可区分但不泄露内部身份信息的业务状态。
- 激活接口受现有速率限制或新增等价限制保护。

## 5. 数据模型

优先复用现有 `ProjectInvitation`、`User`、`Role`、用户角色关联和 `ProjectMember` 模型。

若现有约束不足，迁移应补充：

- 用户邮箱在租户内的规范化唯一约束。
- 项目成员 `(project_id, user_id)` 唯一约束。
- 邀请接受并发所需的行锁或等价原子更新条件。

不新增独立注册令牌表。项目邀请令牌是本流程唯一的注册授权凭证。

## 6. 后端设计

### 6.1 邀请预检

新增无需登录但受令牌保护的接口：

`GET /api/v1/projects/invitations/{token}/preview`

返回：

- `status`: `active`、`expired`、`revoked`、`accepted` 或 `invalid`
- `masked_email`
- `project_name`
- `expires_at`
- `can_activate`
- `can_accept_existing_user`

无效令牌不返回项目名称或邮箱掩码。

### 6.2 新用户激活

新增无需登录但受令牌保护的接口：

`POST /api/v1/projects/invitations/{token}/activate`

请求：

- `full_name`
- `password`

处理流程：

1. 以摘要定位邀请并锁定邀请记录。
2. 校验邀请状态。
3. 规范化邀请邮箱。
4. 查询所有租户范围内的同邮箱用户。
5. 若存在其他租户账号，拒绝激活。
6. 若存在同租户账号，返回需要登录后接受邀请的状态，不重置密码。
7. 创建同租户用户。
8. 查找或创建受控的最低权限项目成员角色，并分配给用户。
9. 创建项目成员关系。
10. 标记邀请已接受。
11. 写入审计和用户通知。
12. 提交事务并返回登录所需结果。

激活成功后返回访问令牌或复用现有登录服务生成会话，使用户可直接进入目标项目。不得在激活成功后要求用户再次输入相同凭据。

### 6.3 现有用户接受

保留当前登录后接受接口，并强化以下行为：

- 使用邀请行锁或原子状态校验。
- 确认登录用户租户和邮箱与邀请一致。
- 幂等处理已存在的项目成员关系。
- 标记邀请接受并写审计与通知。

### 6.4 最低权限角色

最低权限角色应使用稳定系统标识，不依赖中文显示名称判断。权限仅覆盖：

- 读取已加入项目；
- 使用项目内被授权的文档、文件和协作能力；
- 不包含租户成员管理、角色策略管理、Provider、配额、运维或发布权限。

若角色不存在，系统在激活事务中使用幂等方式创建。

## 7. 前端设计

扩展 `/invitations/[token]` 页面为邀请激活入口。

### 7.1 页面状态

- 加载中：校验邀请。
- 有效邀请、未登录：显示项目名称、掩码邮箱、已有账号登录入口和新用户激活表单。
- 有效邀请、已登录且身份匹配：显示接受并进入项目操作。
- 已登录但身份不匹配：解释冲突并允许切换账号。
- 过期：提示联系项目负责人续期。
- 已撤销：提示邀请已失效。
- 已接受：提示使用对应账号登录。
- 无效：不展示项目和邮箱信息。

### 7.2 激活表单

- 邮箱只读显示为掩码值。
- 姓名必填。
- 密码和确认密码必填，并显示密码策略校验结果。
- 提交期间禁用重复操作。
- 成功后保存会话并跳转项目主页。

### 7.3 负责人邀请页面

继续使用现有邀请管理页面，并补充说明：

- 外部新用户可通过链接激活账号。
- 邀请链接属于敏感凭证。
- 续期会使旧链接失效。

## 8. 事务、并发与错误处理

- 激活和接受必须在单个数据库事务内完成。
- 对邀请记录加行锁，防止两个请求同时消费令牌。
- 用户、角色分配或项目成员创建失败时不得标记邀请已接受。
- 唯一约束冲突应转换为幂等结果或明确冲突，不返回数据库内部错误。
- 审计记录与核心事务一致提交。
- 非关键通知失败不得回滚已成功的成员加入，但必须记录可观测错误。

## 9. 审计与通知

至少记录：

- `project.invitation.preview`：仅记录成功或异常状态摘要，不记录明文令牌。
- `project.invitation.activate`
- `project.invitation.accept`
- `project.invitation.activation_rejected`

激活成功后：

- 通知新成员已加入项目。
- 通知项目负责人邀请已被接受。
- 审计元数据包含项目 ID、邀请 ID、用户 ID 和结果，不包含密码或明文令牌。

## 10. 测试策略

### 10.1 后端

- 有效邀请创建新用户、最低权限角色和项目成员。
- 激活成功返回有效会话。
- 无邀请不能注册。
- 过期、撤销、已接受和无效令牌被拒绝。
- 同租户已有账号要求登录接受，不重置密码。
- 跨租户同邮箱被拒绝。
- 弱密码被拒绝。
- 并发激活只有一个成功。
- 任一步骤失败时事务回滚。
- 审计不包含明文令牌或密码。

### 10.2 前端与 E2E

- 新用户完成预检、激活并进入项目。
- 已有用户登录后接受并进入项目。
- 过期、撤销、身份冲突和表单校验状态正确显示。
- 重复点击不会产生重复账号或成员关系。

### 10.3 发布门禁

- 重点 API 测试、身份与项目邀请回归。
- Web typecheck 和生产构建。
- 邀请激活聚焦 Playwright。
- GitHub CI 全部通过。
- 由于涉及身份、权限和数据库事务，发布时执行完整 Release 门禁和生产认证烟测。

## 11. 风险与回滚

### 11.1 主要风险

- 邀请令牌泄露导致未授权账号创建。
- 最低权限角色配置错误导致越权。
- 并发接受产生重复账号或项目成员。
- 跨租户同邮箱错误合并。

### 11.2 控制措施

- 摘要存储、单次消费、过期和撤销控制。
- 稳定的系统角色标识与权限回归测试。
- 行锁、唯一约束和事务。
- 明确拒绝跨租户同邮箱。

### 11.3 回滚

- 回滚应用代码与对应迁移。
- 已激活账号和项目成员属于真实业务数据，不自动删除。
- 必要时由管理员撤销成员关系并停用账号。
- 回滚后保留现有用户登录后接受邀请能力。

## 12. 验收标准

- 外部新用户无需管理员预先创建账号，即可通过有效邀请激活并进入项目。
- 系统仍不允许无邀请注册。
- 新用户仅获得最低权限，不能访问租户管理和运维能力。
- 已有用户、跨租户冲突、过期、撤销、重复和并发场景行为确定。
- 所有身份、成员关系和邀请状态变更可审计。
- 聚焦测试、构建、E2E 和 GitHub CI 全部通过。

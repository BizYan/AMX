# AMX v1.0.0 Release Promotion Blocker Specification

Date: 2026-06-18
Release candidate SHA: `1877e0b4a0cd208890391b76afc8c5f23647cd3b`
Rollback target: `v0.8.8` / `e3808ddaa9519d92d2b4bdd9d238aa7cb177936d`
Status: implemented as an isolated candidate verification gate. Do not tag or deploy from this document.

## Promotion Blocker

`v1.0.0` must remain blocked until the exact release candidate SHA can be verified in an isolated environment that cannot interfere with production.

Current evidence proves:

- current production `v0.8.8` is healthy and authenticated smoke passes;
- `main@1877e0b4a0cd208890391b76afc8c5f23647cd3b` passed CI, Docker Compose config validation, deterministic E2E, and PostgreSQL Alembic migration checks in GitHub Actions.

Current evidence does not yet prove:

- real authenticated smoke against the exact candidate SHA;
- isolated runtime behavior for candidate containers, ports, networks, volumes, and database state.

## Required Isolated Candidate Verification Specification

The next implementation PR must satisfy this specification before owner promotion review can continue. This is not an optional design note. A candidate verification path that omits any mandatory requirement below must not be used as `v1.0.0` pre-tag evidence.

### 1. Docker Compose Project Name

Run the candidate stack with an explicit, unique Compose project name:

```bash
COMPOSE_PROJECT_NAME=amx_rc_1877e0b
docker compose -p amx_rc_1877e0b -f infra/docker-compose.yml --env-file .env.rc.1877e0b up -d --build
```

The project name must be included in all follow-up commands:

```bash
docker compose -p amx_rc_1877e0b -f infra/docker-compose.yml ps
docker compose -p amx_rc_1877e0b -f infra/docker-compose.yml logs --tail=200
docker compose -p amx_rc_1877e0b -f infra/docker-compose.yml down
```

### 2. Fixed `container_name` Values

The current compose file uses fixed `container_name` values for production services. A safe candidate path must remove or parameterize those names before running an isolated stack.

Acceptable design options:

1. remove `container_name` entries and let Compose derive names from `COMPOSE_PROJECT_NAME`;
2. parameterize each name with a release-candidate suffix, for example `${AMX_CONTAINER_PREFIX:-consultant_ai}_api`.

Required candidate container names must not collide with production names:

- production: `consultant_ai_api`, `consultant_ai_web`, `consultant_ai_worker`, `consultant_ai_postgres`, `consultant_ai_redis`;
- candidate example: `amx_rc_1877e0b_api`, `amx_rc_1877e0b_web`, `amx_rc_1877e0b_worker`, `amx_rc_1877e0b_postgres`, `amx_rc_1877e0b_redis`.

### 3. Candidate Env File Isolation

The current compose file has service-level `env_file: ../.env` entries. That is unsafe for candidate verification because `--env-file .env.rc.1877e0b` controls Compose variable substitution, but does not replace service-level `env_file` entries. A candidate stack must never read production `.env`.

The implementation PR must choose exactly one of these approaches:

1. remove service-level `env_file` entries from the candidate compose path and use only an explicit `AMX_ENV_FILE` variable for service env injection;
2. add an `infra/docker-compose.candidate.yml` override that replaces every service-level `env_file` entry with the candidate env file, for example `.env.rc.1877e0b`.

Required behavior:

- candidate `api`, `worker`, and `web` services must read only the candidate env file;
- no service in the candidate stack may read `../.env`;
- rendered candidate `docker compose config` must prove there is no `../.env` reference;
- the candidate env file path must be explicit and must not equal `.env`, `../.env`, `/home/ubuntu/amx/production/AMX/.env`, or any production env path.

### 4. Non-Conflicting Ports, Networks, Volumes, And Database Names

The candidate environment must use unique bind ports, network names, volume names, and database names.

Proposed candidate env:

```dotenv
ENVIRONMENT=staging
COMPOSE_PROJECT_NAME=amx_rc_1877e0b
AMX_RUNTIME_NETWORK=amx_rc_1877e0b_network

POSTGRES_DB=amx_rc_1877e0b
POSTGRES_USER=consultant
POSTGRES_PASSWORD=<candidate-only-password>
DATABASE_URL=postgresql+asyncpg://consultant:<candidate-only-password>@postgres:5432/amx_rc_1877e0b

POSTGRES_BIND_ADDRESS=127.0.0.1
POSTGRES_HOST_PORT=25432
REDIS_BIND_ADDRESS=127.0.0.1
REDIS_HOST_PORT=26379
API_BIND_ADDRESS=127.0.0.1
API_HOST_PORT=28000
WEB_BIND_ADDRESS=127.0.0.1
WEB_HOST_PORT=23000

NEXT_PUBLIC_API_URL=http://127.0.0.1:28000/api/v1
BOOTSTRAP_ADMIN_EMAIL=<candidate-admin-email>
BOOTSTRAP_ADMIN_PASSWORD=<candidate-admin-password>
JWT_SECRET_KEY=<candidate-only-32-byte-or-longer-secret>
```

Candidate volumes must be project-scoped or explicitly named with the candidate suffix:

- `amx_rc_1877e0b_postgres_data`
- `amx_rc_1877e0b_redis_data`

No candidate service may bind to production ports `15432`, `16379`, `18000`, or `3000`.

The PostgreSQL healthcheck must also be environment-driven. Replace hardcoded healthcheck values such as:

```yaml
pg_isready -U consultant -d consultant_ai
```

with:

```yaml
pg_isready -U $$POSTGRES_USER -d $$POSTGRES_DB
```

This is mandatory so candidate database names and users are actually validated.

### 5. Candidate Restart Policy

The candidate environment must set API, worker, and web restart policies to `no`:

```yaml
services:
  api:
    restart: "no"
  worker:
    restart: "no"
  web:
    restart: "no"
```

Candidate verification must fail fast and preserve logs instead of silently restarting release-candidate services.

### 6. Mandatory Pre-Start Safety Assertions

Before `docker compose up`, the implementation must run explicit safety assertions and fail closed if any assertion fails:

```bash
test "$CANDIDATE_ENV_FILE" != ".env"
test "$CANDIDATE_ENV_FILE" != "../.env"
test "$CANDIDATE_ENV_FILE" != "/home/ubuntu/amx/production/AMX/.env"
case "$COMPOSE_PROJECT_NAME" in amx_rc_*) ;; *) exit 1 ;; esac
test "$POSTGRES_DB" != "consultant_ai"
test "$POSTGRES_DB" = "amx_rc_1877e0b"
test "$AMX_RUNTIME_NETWORK" = "amx_rc_1877e0b_network"
test "$API_HOST_PORT" != "18000"
test "$WEB_HOST_PORT" != "3000"
test "$POSTGRES_HOST_PORT" != "15432"
test "$REDIS_HOST_PORT" != "16379"
test "$(pwd)" != "/home/ubuntu/amx/production/AMX"
```

The actual implementation may use a script instead of inline shell, but it must verify:

- candidate env file is not production `.env`;
- Compose project name is isolated and starts with `amx_rc_`;
- ports are non-conflicting;
- network name is isolated;
- volumes are isolated and candidate-scoped;
- database name is isolated;
- working directory is not the production path.

The rendered `docker compose config` must be inspected before startup to confirm:

- no production `container_name` is present;
- no `../.env` service-level env file is present;
- no production bind ports are present;
- candidate network and volume names are present.

### 7. Safe Staging Or Ephemeral CI Deployment Path

Preferred path: ephemeral CI verification for the exact SHA.

The workflow should:

1. checkout `1877e0b4a0cd208890391b76afc8c5f23647cd3b`;
2. generate a candidate-only `.env.rc.1877e0b` from GitHub Actions secrets or protected staging variables;
3. run pre-start safety assertions;
4. render and inspect candidate `docker compose config`;
5. run the isolated Compose project with `COMPOSE_PROJECT_NAME=amx_rc_1877e0b`;
6. runtime-start only `postgres`, `redis`, and `api`;
7. keep `worker` and `web` config-isolated but not runtime-verified by this workflow;
8. run historical migration compatibility baseline verification and authenticated smoke against `http://127.0.0.1:28000`;
9. upload logs, health output, smoke output, migration output, rendered compose config, pre-start assertion output, and `docker compose ps` as artifacts;
10. always run teardown.

Acceptable alternative: an OCI staging slot only if it has its own base path, env file, ports, Compose project name, volumes, and network. It must not reuse production container names or production bind ports.

### 8. Historical Migration Compatibility Baseline Verification

The candidate path must run migration checks against disposable PostgreSQL state, not production data.
This gate is a historical migration compatibility baseline verification, not a clean empty-database full-history migration proof.

Minimum required commands:

```bash
uv run --directory apps/api alembic upgrade head
uv run --directory apps/api alembic downgrade 0021_invitation_delivery
uv run --directory apps/api alembic upgrade head
```

The candidate DB must be isolated and disposable for this verification. The
repository compatibility fixture is:

- create PostgreSQL extensions required by the app;
- provide the minimal legacy `projects` and `documents` baseline tables and ORM
  smoke columns required by current authenticated smoke paths;
- stamp `0021_invitation_delivery`;
- run upgrade, downgrade to `0021_invitation_delivery`, and upgrade again.

This fixture is intentionally documented here because it is broader than the
minimal CI migration fixture and is needed for authenticated candidate smoke.
Clean empty-database full-history migration proof is not claimed by this
candidate workflow.

### 10. Teardown Procedure

Teardown must always run, including after failed tests:

```bash

docker network rm amx_rc_1877e0b_network || true
```

Post-teardown evidence must show no candidate containers, networks, or volumes remain:

```bash

docker network ls --filter name=amx_rc_1877e0b
docker volume ls --filter name=amx_rc_1877e0b
```

## Implemented Verification Path

The implementation path is:

- base Compose keeps production defaults while parameterizing container names and env file path;
- `infra/docker-compose.candidate.yml` supplies candidate-only env files, candidate-scoped volumes, and `restart: "no"` for API, worker, and web;
- `infra/deploy/validate-candidate-verification.sh` performs fail-closed pre-start assertions;
- `.github/workflows/candidate-verification.yml` is manual-only and verifies an exact candidate SHA in an isolated worktree without tag creation, production deployment, OCI production access, or production secrets.

The workflow must not be dispatched automatically. It is a release-owner gate for collecting pre-tag evidence.

## Required Future Changes

Any future change to this path must preserve these implementation boundaries:

Required implementation boundaries:

- parameterize or remove fixed compose container names;
- eliminate candidate reads of production `.env` by either using `AMX_ENV_FILE` only or adding `infra/docker-compose.candidate.yml` to replace every service-level `env_file`;
- make PostgreSQL healthcheck environment-driven with `pg_isready -U $$POSTGRES_USER -d $$POSTGRES_DB`;
- set candidate API, worker, and web restart policies to `restart: "no"`;
- add mandatory pre-start safety assertions;
- add an ephemeral candidate verification workflow or a safe staging slot workflow;
- include secret-safe `.env` generation;
- enforce teardown with `if: always()`;
- upload all evidence artifacts;
- keep production deployment and release tag creation out of the candidate verification workflow.

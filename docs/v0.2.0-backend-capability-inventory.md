# v0.2.0 后端真实能力盘点报告 (Backend Capability Inventory)

本报告详细列出了 ConsultantAIP 系统中前端已调用但后端能力缺失、或者前后端接口契约不一致的模块。通过本次彻底的源码分析，我们为 v0.2.0 后端真实能力补齐打下了坚实的技术基础。

## 核心漏洞与缺陷清单

### 1. Alembic 迁移遗漏 (P0 级别致命缺陷)
后端虽然在模型层定义了 `ProjectInvitation`、`ProjectSettings`、`SourceFile`、`ExportJob` 和 `ExportArtifact` 表，但 **Alembic 迁移文件中完全缺失这些表**（只有 `0001_initial` 和 `0002_partition`）。数据库中这些表并不存在，任何写入都会导致 Relation 不存在的致命报错！

### 2. 模板上传接口参数致命冲突 (P1 级别缺陷)
后端上传模板文件的端点为 `POST /templates/upload?template_id={id}`（接收 Form 文件），而前端 `TemplatesPage` 却往 `POST /templates/{id}/versions` 发送 `FormData`（该接口后端只支持 JSON `TemplateVersionCreate`），导致 100% 触发 422 格式报错瘫痪。

### 3. 文档导出接口不匹配 (P1 级别缺陷)
后端提供了一套完善的 Word/Markdown/PPTX 导出 Service，但前端 `api-client.ts` 声明的 `/exports` 接口与后端完全对不上，且 `/exports/page.tsx` 和文档详情页均采用纯前端 Blob 离线导出 Markdown，根本未调用后端 API。

### 4. Provider 测试连接 422 报错与缺乏安全降级 (P1 级别缺陷)
前端调用测试连接接口 `POST /providers/{id}/test` 时未传 Body，而后端 `data: ProviderTestRequest` 设为必填，导致 422 报错。且后端未实现 Sandbox 安全降级模式，如果测试本地或未配密钥的 Provider 会直接失败。

## 降级与安全边界设计 (Safety & Sandbox Fallback)

1. **项目邀请发送降级**：
   - 不引入复杂的外部邮件发送服务，只在后端保存邀请记录并生成邀请 Token。
   - 前端点击“邀请”后，成功生成邀请链接（如 `http://localhost:3000/invite?token=xxx`）展示给用户，供用户直接复制，完美达成功能闭环。

2. **文档导出格式支持**：
   - 后端目前已完整支持 Markdown 导出。
   - 对于 PDF/DOCX/PPTX 等格式，若本地环境缺少底层依赖（如 LibreOffice 或特定 Python 库），后端服务返回明确的格式不支持或安全降级，前端展示优雅的友好提示，而非静默失败或程序崩溃。

3. **LLM Provider 连接测试 Sandbox 模式**：
   - 为避免在本地 E2E 测试和联调时泄漏真实大模型 API 密钥，在后端 `test_provider` 中建立 Sandbox 检测机制：
     - 若 `api_key` 为 `test_api_key`、`sandbox`、为空，或 Provider 名字包含 "Sandbox" / "Mock"：
     - 开启 **Sandbox Fallback**：不请求第三方真实 LLM Gateway，而是由后端模拟一个延迟 `50ms` 的生成响应并返回成功。
     - 若提供真实 API 密钥，则尝试请求实际接口，同时做好超时与连接异常的健壮性捕捉。

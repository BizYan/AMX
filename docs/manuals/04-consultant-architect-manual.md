# 咨询顾问与解决方案架构师操作手册

## 1. 角色定位

咨询顾问与解决方案架构师负责把原始资料、访谈信息和专业判断转化为结构化知识与可交付文档，并确保需求、设计、接口、数据、测试和结论之间具有可解释关系。

## 2. 推荐工作路径

```text
理解项目范围
  → 上传并核对来源资料
  → 建立知识条目与关系
  → 选择模板和生成策略
  → 对话式分章节编写
  → 执行质量检查和人工修订
  → 建立追溯、处理矛盾和影响
  → 提交评审
  → 根据意见修订并形成基线
  → 导出交付物
```

## 3. 资料管理

进入项目资料区上传来源文件。上传后检查文件状态、名称、用途和可下载性。对于不能解析、内容过时或与项目无关的文件，应及时处理。

资料治理建议：

- 使用能说明来源和版本的文件名；
- 不上传超出合同或授权范围的数据；
- 对客户敏感信息遵循字段权限和项目成员边界；
- 在生成内容前确认资料完整性；
- 删除资料前确认其是否被知识或文档引用。

## 4. 知识治理

### 4.1 创建知识条目

将资料中的事实、需求、约束、决策、风险和术语整理为知识条目。每个条目应表达一个可复核结论，并保留来源。

### 4.2 建立关系

为知识条目建立上下游、依赖、冲突或其他业务关系。利用图谱、邻居和路径分析检查：

- 某需求依赖哪些业务规则；
- 某设计决策影响哪些接口和测试；
- 某风险源于哪些约束；
- 某结论是否缺少证据。

### 4.3 使用来源与血缘

对重要结论检查来源和血缘。没有来源的知识不一定错误，但不得在未经确认时作为正式事实。知识图谱用于提升可解释性，不替代顾问判断。

## 5. 对话式文档编写

### 5.1 选择文档类型

根据交付目标选择 URS、BRD、PRD、用户故事、详细设计、接口文档、数据字典或测试用例。不要用一个文档类型替代所有表达：

- URS/BRD 侧重用户与业务需要；
- PRD/用户故事侧重产品行为和验收；
- 详细设计、接口和数据字典侧重实现约束；
- 测试用例侧重可验证结果。

### 5.2 创建生成会话

在项目文档生成页创建会话，选择文档类型、模板和必要上下文。开始前明确：

- 目标读者；
- 业务范围与不包含范围；
- 使用的来源资料；
- 已确认与待确认事项；
- 质量和交付要求。

### 5.3 分章节推进

系统以章节为单位组织生成过程。每个章节应完成以下动作：

1. 阅读生成草稿并核对事实；
2. 通过对话补充背景、规则和限制；
3. 修正过度推断、空泛表达和无来源结论；
4. 确认章节、继续修改或跳过；
5. 检查章节与前后文是否一致。

不要一次性接受整篇 AI 草稿。分章节确认能够降低错误扩散，并保留更清晰的形成过程。

### 5.4 定稿与人工修订

会话完成后定稿为正式文档，再进入文档编辑与评审流程。定稿并不等于批准。顾问仍需对内容准确性、完整性、可实施性和专业表达负责。

## 6. 质量、版本和基线

系统支持完整性、一致性、MECE 和引用检查。处理质量结果时：

- 将高风险事实错误优先于格式问题；
- 对缺失项补充来源或明确待确认；
- 对矛盾项回到知识和变更链路处理；
- 不为提高分数而加入无业务价值内容。

重要评审点或正式交付前应形成版本或基线。需要恢复历史内容时，先检查影响，再执行回滚或快照恢复。

## 7. 追溯与变更

顾问应主动维护需求、设计、接口、数据、测试和验收关系。发现以下情况时进入变更流程：

- 客户提出的新要求改变已批准范围；
- 业务规则变化影响多个文档；
- 设计调整导致接口或测试变化；
- 文档之间出现矛盾；
- 已发布内容需要实质修订。

使用影响分析和同步建议确认下游对象，不要只修改当前文档。

## 8. 使用 Skill、Agent 和工作流

### 8.1 Skill

Skill 是可复用的任务能力。使用前检查名称、说明、状态、适用文档和测试结果。只有已发布且适用于当前任务的 Skill 才应进入正式工作。

### 8.2 Agent

Agent 是绑定 Skill 的执行配置。运行前确认 Agent 状态、绑定能力、Provider 和目标输出。运行后检查任务、事件和生成结果，失败时依据运行证据处理。

### 8.3 工作流

对跨步骤任务使用工作流。执行前完成 DAG 校验、预览和预检，确认依赖和输入。工作流成功只表示技术执行完成，输出仍需人工检查和业务评审。

## 9. 提交评审和导出

提交评审前检查：

- 内容与项目范围一致；
- 关键结论有来源；
- 待确认事项显式标注；
- 质量问题已处理；
- 追溯关系完整；
- 版本和状态正确。

顾问可按权限执行文档导出，但项目交付包和客户发布应由项目负责人依据门禁统一管理。

## 10. 常见问题

| 现象 | 处理方式 |
|---|---|
| AI 内容看似完整但不准确 | 回到来源和知识条目逐条核对，标记待确认 |
| 章节之间互相矛盾 | 使用矛盾与追溯功能，统一业务规则后修订 |
| 工作流执行成功但结果不可交付 | 执行人工质量与专业审查，不把技术成功当业务通过 |
| 修改后影响范围不明确 | 创建影响分析，检查下游文档和测试 |
| 无法执行 Agent | 检查权限、Agent 状态、Skill、Provider、配额和预检 |

## 11. 顾问与架构师检查清单

- [ ] 来源资料合法、完整、版本明确
- [ ] 关键知识条目具有来源和关系
- [ ] 文档类型和模板符合目标
- [ ] 每个章节经过人工核对
- [ ] 质量问题和待确认事项已处理
- [ ] 追溯、矛盾和影响已检查
- [ ] Agent/工作流输出经过人工验证
- [ ] 提交评审的版本、状态和证据完整

# 业务评审人与审批者操作手册

## 1. 角色定位

业务评审人与审批者负责判断交付内容是否准确、完整、一致、可验证并符合业务目标。该角色应独立检查证据，不应仅依据文档排版、AI 质量分数或作者说明直接批准。

## 2. 主要入口

| 入口 | 用途 |
|---|---|
| `/collaboration` 协同验收 | 查看评审队列、认领工作、通过或退回 |
| `/documents`、项目文档详情 | 阅读文档、评论、线程、版本和状态 |
| `/documents/contradictions` | 处理文档矛盾 |
| 项目追溯页 | 检查覆盖、来源和上下游关系 |
| 项目变更页 | 审批变更和影响 |
| 项目验收页 | 复核交付门禁和客户验收状态 |
| `/audit` | 检查关键操作证据 |

## 3. 评审准备

开始评审前确认：

- 当前评审对象和版本；
- 项目范围、目标读者和验收标准；
- 作者、负责人和截止时间；
- 文档引用的来源资料；
- 已知风险、待确认事项和前次意见；
- 自己是否具有评审或批准权限。

若评审范围或验收标准不清楚，应先要求项目负责人澄清，不应以个人偏好代替项目标准。

## 4. 执行文档评审

### 4.1 认领和阅读

在协同验收中心认领评审事项并标记已读。进入文档详情后，从以下维度审查：

1. **范围正确性**：是否覆盖约定范围，是否混入未授权范围。
2. **事实与来源**：关键结论是否有可信来源，是否存在未经确认的推断。
3. **逻辑一致性**：章节、文档和知识之间是否矛盾。
4. **完整性**：关键流程、角色、异常和约束是否遗漏。
5. **可实施性**：设计、接口、数据和规则是否足够明确。
6. **可验证性**：需求是否能够通过测试或验收证明。
7. **风险与合规**：敏感信息、商业条款、权限和审计要求是否处理。

### 4.2 使用评论和线程

评论应说明问题、影响和期望结果。对需要讨论的问题使用线程；作者处理后，评审人核对修订结果再解决线程。

有效评论示例：

> 第 3.2 节将“审批后生效”写为“提交后生效”，与业务规则和测试用例 TC-17 不一致。请统一规则，并评估接口状态机和相关测试影响。

避免仅写“有问题”“再完善”等不可执行意见。

### 4.3 使用快照与锁

复杂修订前可保留快照。发现并发编辑风险时检查文档锁状态。恢复快照前应确认会覆盖哪些变化，并通知相关作者。

## 5. 通过、退回和批准

### 5.1 退回修订

出现以下情况应退回：

- 核心事实错误或无来源；
- 关键范围、流程或异常遗漏；
- 文档之间存在未解决矛盾；
- 追溯覆盖不足；
- 高风险意见未处理；
- 当前版本不具备正式放行条件。

退回时写清必须修复项和再次提交条件。

### 5.2 通过评审

通过评审表示当前评审要求已满足，不必然等于最终批准或发布。通过前检查所有阻断意见已经解决，且审查的是最新版本。

### 5.3 批准

批准是高风险动作。批准前确认：

- 当前版本和基线正确；
- 评审意见闭环；
- 追溯和质量证据可接受；
- 变更影响已处理；
- 批准权限有效；
- 不存在应由客户、法务或项目负责人决定的未决事项。

## 6. 变更与影响审批

对变更请求重点审查：

- 变更原因是否成立；
- 是否改变合同或批准范围；
- 对文档、设计、接口、数据、测试和交付日期的影响；
- 风险、成本和回滚方案；
- 是否需要客户确认；
- 应用后是否更新追溯和验收证据。

对字段补丁也应按相同原则审查，不能因为修改范围小就忽略业务影响。

## 7. 追溯与矛盾处理

使用追溯矩阵检查：

- 需求是否有来源；
- 需求是否被设计和测试覆盖；
- 测试是否对应明确需求；
- 验收条件是否可以证明目标；
- 变更是否同步到所有下游对象。

处理矛盾时，不要简单选择文字更完整的一方。应回到权威来源、业务决策和批准基线，确认正确规则后再同步修订。

## 8. 验收复核

在项目交付前，评审人应协助项目负责人复核：

- 必需文档和批准状态；
- 高风险评论和变更；
- 追溯覆盖；
- 交付包内容；
- 客户验收条件；
- 整改项闭环；
- 审计与回执证据。

技术准备度、自动质量分数和导出成功都不能单独替代业务验收。

## 9. 评审检查清单

- [ ] 评审对象、版本和标准明确
- [ ] 核心事实和来源已核对
- [ ] 范围、逻辑、完整性和可验证性已检查
- [ ] 评论具体、可执行并已闭环
- [ ] 矛盾、追溯和变更影响已复核
- [ ] 通过、退回或批准决定有证据支持
- [ ] 未把 AI 分数或技术成功误当业务通过

# 项目负责人操作手册

## 1. 角色定位

项目负责人对项目范围、团队协作、交付计划、文档门禁、客户验收和正式关闭负责。该角色的核心工作不是亲自完成所有内容，而是确保每个阶段都有负责人、证据、状态和明确的放行条件。

## 2. 主要入口

| 入口 | 用途 |
|---|---|
| `/dashboard`、`/delivery` | 查看跨项目进度、风险和待办 |
| `/projects` | 创建、启动、归档和恢复项目 |
| `/projects/{id}` | 项目驾驶舱 |
| `/projects/{id}/plan` | 交付计划和里程碑 |
| `/projects/{id}/members` | 项目成员与邀请 |
| `/projects/{id}/documents` | 文档组合与状态 |
| `/projects/{id}/traceability` | 追溯与覆盖 |
| `/projects/{id}/changes` | 变更请求与影响 |
| `/projects/{id}/acceptance` | 内部和客户验收、交付关闭 |
| `/projects/{id}/settings` | 项目和文档生命周期设置 |
| `/exports` | 交付导出与产物下载 |

## 3. 启动项目

### 3.1 使用项目启动向导

进入“项目文档”，选择启动项目。建议在创建前准备：

- 项目名称与业务目标；
- 合同或约定范围；
- 项目负责人；
- 初始成员；
- 需要产出的文档类型；
- 适用模板或启动蓝图；
- 关键里程碑和验收条件。

启动向导可根据蓝图初始化项目、文档、成员和启动计划。提交后进入项目驾驶舱检查初始化结果。若启动计划存在失败项，应先处理并使用重试功能，不要在基础初始化不完整时直接推进交付。

### 3.2 建立项目成员

在项目成员页加入租户成员或发送项目邀请。对每名成员明确：

- 项目职责；
- 应交付内容；
- 是否具有管理或评审责任；
- 加入与退出时间；
- 是否涉及敏感信息。

项目成员关系与租户角色共同决定访问能力。项目结束后应及时移除不再需要访问的成员。

## 4. 建立交付计划

初始化交付计划后，创建或调整里程碑。每个里程碑应包含可验证的结果，而不是笼统活动。

推荐里程碑结构：

1. 资料收集与范围确认；
2. 知识治理和需求基线；
3. 核心文档编写；
4. 评审、修订与批准；
5. 追溯与交付准备；
6. 客户验收与关闭。

项目负责人可排序、启动、完成或重新打开里程碑。完成前应检查依赖、负责人、交付物和门禁，避免只更新状态而没有实际证据。

## 5. 管理文档生产

### 5.1 文档组合

在项目文档页确认本项目需要哪些文档。系统支持 URS、BRD、PRD、用户故事、详细设计、接口文档、数据字典和测试用例。

对每份文档明确：

- 负责人和评审人；
- 所用资料、知识和模板；
- 目标状态；
- 截止时间；
- 是否为交付包必需项；
- 是否需要客户确认。

### 5.2 生命周期治理

项目设置可控制文档生命周期。项目负责人应确保文档按照草稿、编写、待评审、评审、修订、批准、发布和归档的治理逻辑推进。

原则：

- 未经评审的内容不得作为正式交付依据；
- 被退回的文档必须完成修订并重新提交；
- 已批准内容发生实质变化时，应通过变更流程处理；
- 发布和归档应由具备相应权限的角色执行。

## 6. 管理评审、变更和追溯

### 6.1 评审协同

在协同验收中心检查评审队列和工作项。对逾期、无人认领、被退回或反复修订的事项及时调整负责人和优先级。

### 6.2 变更管理

对影响范围、合同、交付日期或已批准内容的修改，应创建变更请求。变更处理应包含：

1. 变更原因；
2. 影响对象；
3. 风险和工作量；
4. 审批意见；
5. 应用结果；
6. 相关文档和测试同步。

### 6.3 追溯检查

使用追溯矩阵检查需求、设计、接口、测试和验收之间的覆盖关系。重点处理：

- 无来源的结论；
- 无实现或无测试覆盖的需求；
- 发生变更但下游未同步的对象；
- 相互矛盾的文档内容；
- 影响分析未处理的高风险项。

## 7. 导出与交付

### 7.1 交付准备

在导出前完成：

- 必需文档齐全；
- 文档状态满足门禁；
- 质量问题和评审意见已闭环；
- 追溯覆盖符合约定；
- 关键里程碑已完成；
- 客户交付范围明确。

### 7.2 创建导出

在导出中心选择项目、文档、格式和变量，创建项目交付包或单项文档导出。任务异步执行，可查看状态、失败信息、重试和产物下载。

若交付包被阻止，应根据准备度提示处理缺失文档、占位内容、未批准内容或其他门禁问题，而不是绕过门禁。

## 8. 客户门户与验收

### 8.1 创建门户链接

在项目验收区创建客户交付门户链接，设置有效期和客户信息。链接应通过安全渠道交付，必要时可撤销并重新创建。

### 8.2 处理客户反馈

客户可下载限定产物并提交接受、待处理或拒绝意见。项目负责人应：

- 核对提交人身份与项目约定；
- 对待处理或拒绝项建立整改责任；
- 跟踪系统生成或关联的协同工作项；
- 完成整改后邀请客户重新验收；
- 保留下载和验收回执证据。

### 8.3 正式关闭与重新打开

只有当内部交付门禁、客户验收和整改均满足条件时，才执行正式关闭。发现遗漏或新增合同义务时，可重新打开交付，但必须记录原因和后续计划。

## 9. 常见风险

| 风险 | 控制方式 |
|---|---|
| 文档很多但没有统一目标 | 用项目范围、文档组合和里程碑约束 |
| 状态显示完成但证据不足 | 检查产物、评审、追溯和验收记录 |
| 客户下载了错误版本 | 只通过受控门户暴露完成产物，撤销过期链接 |
| 变更没有同步下游 | 使用影响分析、追溯和同步建议 |
| 项目成员权限过大 | 与管理员共同复核角色、项目成员和字段权限 |

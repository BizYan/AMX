# 平台与租户管理员操作手册

## 1. 角色定位

平台与租户管理员负责建立系统可控运行的基础条件：成员身份、角色权限、敏感字段、服务端凭据、Provider、外部集成、配额和治理证据。该角色不应代替项目负责人批准业务内容，也不应代替运维负责人执行未经授权的生产发布。

## 2. 主要入口

| 入口 | 用途 |
|---|---|
| `/team` 团队权限 | 成员、角色、ABAC、字段权限、权限模拟和审计 |
| `/settings` 设置 | 租户成员、角色和服务端 API Key |
| `/providers` Provider | 模型与能力供应方配置、连接测试和健康 |
| `/integrations` 集成中心 | 外部系统连接、同步、Webhook 和投递 |
| `/quotas` 配额管理 | 使用量、限流和风险控制 |
| `/audit` 审计中心 | 检索关键操作证据 |
| `/system-health` 运维监控 | 查看能力准备度和平台风险 |

## 3. 建立租户权限体系

### 3.1 初始化标准角色

进入“团队权限”，检查标准角色是否存在。系统提供交付管理员、项目负责人、咨询顾问、业务评审人和平台运维负责人模板。

操作原则：

1. 优先用标准角色覆盖常见岗位。
2. 仅在职责确有差异时创建自定义角色。
3. 按最小必要权限勾选项目、文档、智能编排和团队动作。
4. 将“批准、发布、授权、运行管理”等高风险动作限制在少数岗位。
5. 定期复核包含通配符权限、团队授权或文档发布权的角色。

### 3.2 创建和分配成员

在成员区创建租户成员，可立即分配初始角色，也可暂不分配。

完成后检查：

- 账号是否启用；
- 初始角色是否符合职责；
- 是否需要加入具体项目；
- 是否意外获得高权限；
- 初始凭据是否通过安全渠道交付并及时更换。

租户角色不自动等于项目成员资格。成员需要参与某个项目时，应由项目负责人在项目成员页加入或邀请。

### 3.3 配置 ABAC 策略

ABAC 用于在角色权限之上，根据上下文执行更精细的允许或拒绝判定。

建议流程：

1. 明确策略目的、作用对象和生效条件；
2. 优先使用拒绝策略保护高风险场景；
3. 保存后使用权限模拟器验证允许与拒绝场景；
4. 检查策略是否误伤正常角色；
5. 在审计记录中确认策略变化已留痕。

拒绝策略通常优先于普通角色授权。出现“角色已有权限但操作仍被拒绝”时，应优先检查 ABAC。

### 3.4 配置字段级权限

字段权限用于控制敏感业务信息。当前可治理的典型字段包括：

- 文档：商业条款、客户联系人、风险评估；
- 项目：项目预算、合同范围；
- Agent 运行：Provider 调用载荷。

可选权限为可读、可写或不可见。配置后必须使用权限模拟器验证实际判定，不应仅依据界面勾选结果判断。

### 3.5 运行权限模拟

权限模拟调用真实后端权限判定器，并显示：

- 最终允许或拒绝结果；
- 判定原因；
- 角色证据；
- 策略证据；
- 字段控制证据。

建议至少模拟以下场景：

- 顾问能编写文档但不能批准；
- 评审人能评审和批准但不能修改团队权限；
- 项目负责人能管理项目但不能修改平台 Provider；
- 无项目关系成员不能访问受限项目；
- 敏感字段对非授权角色不可见。

模拟结果会形成审计事件，可作为上线验收证据。

## 4. 管理服务端 API Key

在设置页创建 API Key 时，遵循以下要求：

1. 使用明确名称标识调用系统和负责人；
2. 只授予所需 Scope；
3. Token 首次显示时通过安全方式交付；
4. 不在文档、代码、工单或聊天中明文保存；
5. 定期轮换并撤销不再使用的 Key；
6. 在审计中心检查异常使用。

API Key 用于服务间访问，不应替代普通用户登录。

## 5. Provider 管理

### 5.1 创建与配置

在 Provider 控制中心新建供应方并填写必要连接配置。凭据应通过受控配置或密钥机制提供，不应写入仓库。

### 5.2 上线前检查

- 连接测试成功；
- Provider 健康状态正常；
- 所需能力和版本已登记；
- 超时、失败和回退行为符合预期；
- 运行记录可查询；
- 敏感调用载荷受到字段权限保护；
- 配额和成本边界明确。

### 5.3 故障处理

当 Provider 异常时：

1. 查看健康和最近运行记录；
2. 区分凭据、网络、限流、模型能力和请求格式问题；
3. 必要时暂停使用或切换回退；
4. 检查受影响 Agent 运行；
5. 保留故障和恢复证据。

## 6. 外部集成管理

集成中心支持供应方配置、连接测试、项目绑定、同步预览、同步运行、失败重试、Webhook、入站事件、投递记录和 Outbox 发布。

上线一个集成的建议步骤：

1. 创建集成 Provider；
2. 执行连接测试；
3. 绑定到限定项目；
4. 先执行同步预览，核对对象范围和字段映射；
5. 再执行正式同步；
6. 检查同步运行、入站事件和投递状态；
7. 对失败事件分析原因后再重试；
8. 确认停用、撤销和数据回收方案。

不要把“Provider 已创建”解释为“第三方业务已完成联调”。必须以成功同步、事件投递和业务核对为准。

## 7. 配额、审计与治理

管理员应持续关注：

- 用户、Provider 和自动化运行的使用量；
- 配额接近上限的趋势；
- 限流、熔断和告警；
- 高权限角色和拒绝策略；
- 被撤销或长期未使用的 API Key；
- 权限模拟和敏感操作审计；
- 能力准备度、激活和验收证据。

## 8. 常见问题

| 现象 | 优先检查 |
|---|---|
| 用户能看到页面但操作被拒绝 | RBAC 动作、ABAC 拒绝策略、字段权限、项目成员关系 |
| 用户有角色但看不到项目 | 是否被加入项目、项目是否归档、租户是否一致 |
| 集成测试成功但没有业务数据 | 项目绑定、同步预览、字段映射、正式同步运行 |
| Agent 运行失败 | Provider 健康、配额、Agent 配置、工作流预检、运行事件 |
| 权限变更无法解释 | 权限模拟结果和审计中心 |

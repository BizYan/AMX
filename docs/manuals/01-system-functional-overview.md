# AMX 系统功能综述

## 1. 文档目的

本文面向企业管理者、咨询交付负责人、产品负责人、解决方案架构师、平台管理员和实施团队，说明 AMX 当前代码中已经形成闭环的业务能力、系统结构、角色分工、关键控制点及适用边界。

AMX 不是单纯的 AI 文档生成器。它将咨询项目中的资料治理、知识结构化、文档编写、评审审批、变更追溯、智能编排、交付导出、客户验收和平台运维纳入同一套可审计工作流。

## 2. 产品定位与价值

AMX 的核心目标是把“依赖个人经验、文件散落、生成过程不可追溯”的咨询交付，转化为“项目化、结构化、可协作、可审计、可复用”的生产过程。

系统主要解决五类问题：

1. **交付过程不透明**：通过项目驾驶舱、里程碑、工作项、评审和验收状态展示真实进度。
2. **资料与结论脱节**：通过来源文件、知识条目、知识链接、文档引用和追溯矩阵建立证据链。
3. **AI 生成不可控**：通过模板、Skill、Agent、工作流、质量检查、人工确认和运行记录约束生成过程。
4. **交付物版本混乱**：通过文档状态、版本、基线、快照、变更请求和导出任务保留历史。
5. **发布与运维缺少证据**：通过审计日志、运行监控、配额、Provider 健康、发布工作流和生产校验建立控制面。

## 3. 总体架构

```text
浏览器 / 客户交付门户
          |
          v
Next.js Web 工作台
          |
          v
FastAPI 业务 API
  |       |        |
  v       v        v
PostgreSQL/pgvector Redis/ARQ Worker 文件与导出存储
          |
          v
LLM Provider / 外部集成
          |
          v
OCI + Nginx + Docker Compose
```

### 3.1 业务边界

- **租户**是组织、用户、角色、策略、配额和平台配置的隔离边界。
- **项目**是咨询交付的主要业务容器，承载成员、资料、知识、文档、计划、变更、验收和导出。
- **文档**是交付内容的主要生产单元，支持章节化编写、质量检查、版本、基线和状态流转。
- **Agent 运行**是受治理的自动化执行单元，保留任务、事件、状态和错误证据。

### 3.2 授权边界

系统权限由多层机制共同决定：

- RBAC：角色授予项目、文档、智能编排和团队管理等资源动作。
- ABAC：按用户、资源、动作和上下文执行允许或拒绝策略。
- 字段级权限：对商业条款、客户联系人、风险评估、项目预算、合同范围和 Provider 调用载荷等敏感字段控制可读、可写或不可见。
- 项目成员关系：即使拥有租户级角色，部分项目操作仍要求项目成员或项目所有者身份。
- 审计：关键授权、模拟、变更和操作写入审计记录。

## 5. 核心业务闭环

### 5.1 项目启动与治理

项目可通过启动向导和蓝图创建。启动过程不仅创建项目记录，还可初始化项目文档、成员和启动计划。项目负责人随后可：

- 查看项目驾驶舱和启动计划；
- 配置项目成员与邀请；
- 初始化交付计划；
- 新建、排序、启动、完成或重新打开里程碑；
- 配置项目设置及文档生命周期策略；
- 归档、恢复或删除项目。

项目启动失败时，系统保留启动计划及失败状态，并提供重试入口，避免出现“项目已创建但初始化过程不可解释”的情况。

### 5.2 资料与知识治理

项目资料区支持文件上传、确认、列表、下载和删除。资料进入系统后，可用于知识提取和后续文档生成。

知识模块支持：

- 知识条目增删改查和批量创建；
- 知识关系建立与维护；
- 搜索、邻居关系、路径分析和图谱工作台；
- 来源、血缘和证据追踪；
- 从来源材料摄取知识；
- 生产准备度、激活与验收证据。

知识图谱的价值不在于绘图本身，而在于让需求、结论、风险和来源之间形成可检查关系。

### 5.3 对话式文档生产

系统支持以下文档类型：

- 用户需求规格说明书（URS）
- 业务需求文档（BRD）
- 产品需求文档（PRD）
- 用户故事
- 详细设计文档
- 接口文档
- 数据字典
- 测试用例

文档可从项目文档区创建，也可进入对话式生成会话。对话式生成支持：

1. 创建生成会话并选择文档类型、模板和上下文；
2. 按章节推进，记录章节状态；
3. 通过消息补充业务背景、约束和修改要求；
4. 确认、跳过或继续章节；
5. 完成后定稿为正式文档；
6. 对生成结果执行完整性、一致性、MECE 和引用检查；
7. 进入评审、修订、批准、发布或归档状态。

文档还支持版本、基线、回滚、引用、影响分析和协作快照。系统因此能够保存“内容为什么这样形成、经过哪些修改、何时被批准”的证据。

### 5.4 评审、变更与追溯

协同验收中心提供评审队列与工作项看板。评审人可认领任务、标记已读、通过验收或退回修订。文档协作链路还包括：

- 评论、讨论线程和评论解决；
- 文档锁、刷新和释放；
- 文档快照及恢复；
- 评审状态流转；
- 变更请求提交、审批、拒绝、应用和取消；
- 字段补丁审批；
- 影响分析与同步建议；
- 矛盾识别和解决；
- 追溯矩阵、覆盖率、建议和验收。

该机制用于降低“修改了一个需求，却没有同步设计、测试和交付物”的风险。

### 5.5 智能编排闭环

智能编排由 Skill、Agent、工作流和运行记录四层组成：

- **Skill**：可复用能力单元，支持创建、发布、停用和测试。
- **Agent**：具备明确配置和 Skill 绑定的执行角色，支持启用与停用。
- **工作流**：以定义和版本描述步骤关系，支持 DAG 校验、预览、预检和执行。
- **运行记录**：保留运行、任务、事件、状态、错误、取消、重试和控制证据。

Worker 使用队列执行异步任务。运维人员可从 Agent 运行中心检查失败原因和重试。Agent 无权绕过文档审批、交付验收或生产发布门禁。

### 5.6 导出、客户验收与正式关闭

导出中心支持项目交付包及 Word、Markdown、PPTX 等产物。导出为异步任务，具有任务状态、产物记录、失败重试和下载入口。

项目交付包受到准备度门禁约束。存在占位内容、缺失必需文档、未批准文档或未完成要求时，系统可以阻止交付包生成。

项目负责人可创建受控的客户交付门户链接。客户通过链接：

- 查看项目和交付包准备状态；
- 查看限定的交付产物；
- 下载允许下载的完成产物；
- 查看验收条件和整改状态；
- 提交或重新提交验收意见；
- 获得验收回执。

门户 Token 只保存哈希，可设置有效期并撤销。客户拒绝或待处理的验收项会形成后续协同工作项；接受后可关闭对应事项。完成内部门禁和客户验收后，项目负责人可正式关闭交付，也可在必要时重新打开。

## 6. 平台治理能力

### 6.1 团队与权限

系统内置可初始化的标准角色模板：

- 交付管理员
- 项目负责人
- 咨询顾问
- 业务评审人
- 平台运维负责人

角色模板只是起点。管理员可以创建自定义角色，并按项目、文档、智能编排和团队权限选择具体动作。权限中心还支持：

- 用户创建、启停、角色分配；
- 角色权限矩阵；
- ABAC 允许/拒绝策略；
- 字段级权限；
- 权限诊断；
- 使用真实后端判定器进行权限模拟；
- 权限生产准备度、激活与验收证据；
- 权限相关审计。

### 6.2 Provider、集成和配额

Provider 控制中心用于管理 LLM 或能力供应方，包括连接测试、版本、能力、运行记录、健康状态和回退信息。

集成中心提供外部系统连接、项目绑定、同步预览、同步运行、失败重试、Webhook、入站事件、投递记录和 Outbox 发布能力。它是一套受治理的集成框架，并不意味着所有第三方产品都已开箱即用。

配额和运维模块支持使用量、限流、熔断、告警和运行风险控制，用于防止成本或故障无限扩散。

## 7. 角色与职责矩阵

| 角色 | 主要责任 | 关键放行权 |
|---|---|---|
| 平台/租户管理员 | 组织、账号、权限、Provider、集成和配额治理 | 授权与平台配置 |
| 项目负责人 | 项目范围、计划、成员、交付和客户验收 | 项目门禁与正式关闭 |
| 咨询顾问/架构师 | 资料、知识、文档、追溯和自动化执行 | 内容提交与专业判断 |
| 业务评审人/审批者 | 评审、退回、批准、变更与风险复核 | 文档批准与验收建议 |
| 外部客户 | 查看限定交付物并提交验收 | 客户验收意见 |
| 运维/发布负责人 | 系统运行、故障处置、发布、部署和回滚 | 技术发布与生产恢复 |

## 8. 状态、门禁与证据

### 8.1 文档生命周期

文档状态包括草稿、编写中、待评审、评审中、需修订、已批准、已发布和已归档。状态流转应以角色权限和业务证据为前提。

### 8.2 交付门禁

正式交付前至少应检查：

- 必需文档是否齐全；
- 文档是否达到要求状态；
- 质量问题是否处理；
- 追溯覆盖是否达到约定标准；
- 里程碑是否完成；
- 导出任务和产物是否成功；
- 客户验收和整改是否闭环；
- 审计证据是否可查询。

### 8.3 生产发布门禁

完整版本发布应满足：

1. 代码通过本地聚焦验证和 GitHub CI；
2. 已接受的 PR 合并到目标分支；
3. 创建语义化版本标签并形成 GitHub Release；
4. 通过 GitHub Actions 部署到 OCI；
5. 生产健康检查、登录与关键路径 Smoke 成功；
6. OCI 部署提交与请求版本一致；
7. GitNexus 服务健康并完成索引刷新；
8. 必要时保留回滚标签或 SHA。

## 9. 安全与审计原则

- 密钥、Token、私钥和生产 `.env` 不进入仓库。
- 客户门户 Token 使用哈希存储并支持撤销和过期。
- 服务端 API Key 应采用最小范围并定期撤销轮换。
- 高权限角色、拒绝策略和字段级权限应定期复核。
- Provider 调用载荷、商业条款、项目预算等敏感字段应按岗位限制。
- 关键业务动作、权限模拟、运行和变更应保留审计证据。
- 对外发布前应执行秘密扫描、依赖检查和生产配置检查。

## 10. 当前能力边界

以下边界在使用和实施时必须明确：

- AI 生成、质量分数和建议是辅助证据，不等于专业审查通过。
- 系统具备评论、线程、锁和快照，但不应描述为多人实时无冲突共同编辑器。
- 集成中心提供治理框架；具体外部系统是否可用，取决于 Provider 配置、凭据和适配实现。
- 权限由后端综合判定。看到页面、按钮或导航不代表拥有操作权限。
- “生产准备度”“激活”“验收证据”用于技术与治理检查，不替代客户合同验收。
- GitNexus 是研发侧代码关系与影响分析工具，不是业务用户功能，也不替代测试和代码审查。

## 11. 建议的标准使用路径

```text
管理员建立组织与权限
        ↓
项目负责人用蓝图启动项目、配置成员与计划
        ↓
顾问上传资料、治理知识、对话式生成并编写文档
        ↓
评审人审阅、退回/批准，团队处理变更和追溯
        ↓
项目负责人执行交付门禁并生成交付包
        ↓
客户通过受控门户下载并提交验收
        ↓
项目负责人关闭交付，审计与运维保留证据
```

## 12. 实施建议

首次上线不应直接追求所有功能同时启用。推荐按以下顺序建立组织能力：

1. 先建立角色、最小权限和项目成员规则；
2. 再统一项目蓝图、模板、文档状态和验收标准；
3. 形成资料、知识、文档、评审、追溯、导出的人工闭环；
4. 在人工闭环稳定后引入 Skill、Agent 和工作流自动化；
5. 最后接入外部系统、配额、生产治理和规模化运营。

该顺序能够避免用自动化放大尚未标准化的业务过程。

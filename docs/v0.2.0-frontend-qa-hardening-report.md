# Frontend QA Hardening & Regression Report (v0.2.0 前端质量加固与回归报告)

为了确保 ConsultantAIP v0.2.0 前端系统的发布质量与多维度稳定性，我们在专属加固分支 `feature/v0.2.0-frontend-qa-hardening` 上对全站路由进行了深度的遍历回归测试。本分支通过引入高强度的 Playwright 回归测试套件，彻底加固了前端不可达、白屏以及中文化质量控制的拦截能力。

---

## 📊 自动化回归测试执行情况

本分支新增并全量调优了专用于全路由回归检测的自动化脚本 `tests/e2e/playwright/frontend-regression.spec.ts`。

### 1. 测试运行环境与指令
* **API 服务端点 (E2E_API_URL)**：`https://amx.yuanda.win/api/v1`
* **运行命令**：
  ```bash
  pnpm --dir apps/web exec playwright test tests/e2e/playwright/frontend-regression.spec.ts
  ```

### 2. 测试断言加固说明
为了彻底杜绝“测试放水”或“低质量通过”，本分支对回归测试的可用性断言进行了如下深度加固：
* **更具体的页面可用性断言**：不仅检测 body 结构，还定义了具体的可见性约束，强制断言主页面区域（`main`、`#main-content`、`.container`、`h1` 等主渲染容器）必须处于 `toBeVisible()` 状态，彻底过滤“虽然生成了 body 但内部白屏/隐藏”的异常。
* **白屏与异常拦截**：严格过滤并拦截 `Application error`、`client-side exception`、`This page could not be found` 以及 `ChunkLoadError`。
* **绝对无英文 Fallback 的中文化验证**：移除了先前版本中所有的英文 Fallback 兼容匹配正则（彻底去掉对 `Agent`、`Template`、`Export`、`Workflow`、`LLM` 等词汇的放行），仅保留严格的纯中文正则断言（如 `/智能体/i`、`/模板/i`、`/导出/i`、`/工作流/i`、`/供应商/i`）。任何残留未汉化的一级标题都将精准触发测试红灯。

## 🛠️ 问题清单与改动事实归类

本着“事实第一”的口径原则，我们将本分支涉及的所有代码、测试发现以及基线状况梳理如下：

### 一、 本分支实际修复/改动的代码
* **无**
  * *事实说明*：本分支作为前端回归与质量加固分支，**没有修改任何前端业务逻辑代码**（包括先前报告中所误称的 `apps/web/src/app/(app)/projects/page.tsx`）。
  * *改动文件*：本分支实际仅有以下两个文件的改动，无任何业务逻辑侵入：
    1. [frontend-regression.spec.ts](file:///C:/ConsultantAIP_antigravity/apps/web/tests/e2e/playwright/frontend-regression.spec.ts) —— 新增端到端自动化回归测试用例。
    2. [v0.2.0-frontend-qa-hardening-report.md](file:///C:/ConsultantAIP_antigravity/docs/v0.2.0-frontend-qa-hardening-report.md) —— 本质量报告文件。

### 二、 本分支新增的测试
* **E2E 路由回归测试 (frontend-regression.spec.ts)**
  * **R.1 一级全量配置页面回归**：自动登录后，深度遍历全站 16+ 个一级页面，严密检查异常崩溃并执行严格纯中文标题断言。
  * **R.2 设置页邀请表单交互回归**：点击“发送邀请”弹出对话框，测试非法邮箱格式时的表单禁用拦截及后端错误 Toast 响应兼容性（确保防污染、无危险物理操作）。
  * **R.3 动态项目路由与子路由回归**：脚本支持自适应抓取系统中渲染的第一个真实 `projectId`，并在项目存在时遍历 9 大子路由（概览、文件、文档、生成、知识库、成员、设置、追溯、变更历史）。本次集成验证运行时未在 E2E 页面中发现可见项目卡片，因此实际执行的是“无项目数据时创建项目弹窗与空状态”分支；项目子路由遍历能力已写入测试，但需要在有可见项目数据的环境中继续验证。
  * **稳定性修正**：集成审查时将页面等待策略从 `networkidle` 调整为 `domcontentloaded` + 可见正文断言，避免被健康检查、轮询或长请求误判为超时；同时将用例超时提升到 120 秒，以适配远端 API 登录偶发慢响应。

### 三、 基线已有能力确认
* **项目卡片“归档项目”Toast 反馈 (人工代码审查确认)**
  * *事实说明*：在 `/projects` 项目列表页中，项目卡片下拉菜单的“归档项目”按钮已由当前开发基线 `feature/v0.2.0` 提供 Toast 提示：“归档功能将在 v0.2.x 中开放，当前版本请先通过项目设置维护项目状态”。
  * *结论*：此交互为当前开发基线已有能力，非本分支代码所写。本分支新增的 E2E 回归测试尚未覆盖项目卡片下拉菜单的归档按钮点击行为，后续如需防止该交互退化，应补充专门的菜单交互测试。

### 四、 仍需中文化问题
* **一级路由主标题未发现残留**
  * *事实说明*：`frontend-regression.spec.ts` 已移除针对 `Agent`, `Template`, `Export`, `Workflow` 等英文标题的 fallback，并对一级路由的可见正文执行中文关键字断言。测试通过说明本轮覆盖到的主标题和关键业务区域未发现英文残留。
  * *范围边界*：该测试不等同于全站所有按钮、表格字段、空状态、错误提示和二级弹窗的完整中文审计。后续仍应通过页面交互测试和人工审查持续补齐细粒度中文化。

### 五、 仍需后端支持功能
* **物理项目归档与物理删除功能 (Pending)**
  * *事实说明*：当前前端通过 Toast 交互成功完成了对“归档项目”的用户操作反馈。但底层的项目状态物理变更以及物理删除 API 尚未在当前后端版本中实现。待后续版本后端 API 就绪后，前端可直接移除提示并接入 API 触发实际归档。

### 六、 暂未覆盖与未处理项
* **项目动态路由白屏防御补丁 (未加装)**
  * *事实说明*：测试脚本已经具备项目存在时遍历 9 大项目子页面的能力，但本次集成验证运行时未发现可见项目卡片，因此没有实际覆盖项目子路由。当前未加装额外业务层防御补丁；后续需要在含项目数据的环境中持续运行该测试，或补充稳定的测试项目夹具。

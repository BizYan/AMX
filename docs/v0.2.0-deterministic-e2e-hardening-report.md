# v0.2.0 前端确定性 E2E 测试夹具与回归加固交付报告

本报告用于记录在专属大任务分支 `feature/v0.2.0-deterministic-e2e-hardening` 下开展的自动化回归测试与 Mock 夹具开发成果。本轮工作通过引入隔离的 Playwright API 拦截机制，实现了不受外界真实数据干扰、稳定可靠的确定性测试用例，并顺手修复了 1 个真实的前端白屏崩溃缺陷。

---

## 一、新增自动化测试与 Mock 夹具文件

为了避免端到端测试对真实后端的物理依赖，本分支共设计并新增了以下测试夹具及代码文件：

1. **Mock 数据源文件**：
   - `apps/web/tests/e2e/playwright/fixtures/mock-data.ts`
   - *覆盖范围*：包含了符合前端 API 契约的租户 (tenant)、管理员用户 (currentUser)、项目 (project)、2个文档 (urs/prd)、源文件 (sourceFiles)、知识节点与关联 (knowledgeNodes/Edges)、说明书模板及版本 (templates/versions)、智能体运行 (agentRuns)、工作流 (workflows)、运维健康与配额监控 (healthStatus/quotas/rateLimits/usageStats) 等 10 多种核心高保真业务假数据。

2. **API Mock 拦截注册器**：
   - `apps/web/tests/e2e/playwright/fixtures/api-mocks.ts`
   - *覆盖范围*：基于 Playwright 的 `page.route()` 路由拦截机制，拦截了后端共计 20+ 个核心 API 接口，包括用户权限、项目管理、文档读写与智能生成、知识图谱查询、模板与供应商管理、智能体工作流执行、监控配额等所有高频请求路径，为 E2E 测试铺设了稳健的本地假接口沙箱。

3. **确定性自动化回归用例**：
   - `apps/web/tests/e2e/playwright/deterministic-regression.spec.ts`
   - *覆盖范围*：包含了 6 大高集成度的功能回归用例，通过引入异步等待 `expect(locator).toContainText()` 机制解决 Next.js 路由过渡瞬间的竞态问题，并优化了表单交互及 Dialog 遮罩的 Escape 释放动作。

## 三、修改的业务代码与发现的前端 Bug 修复

本分支严格遵循用户权限规范，**未触碰任何后端代码目录**，仅在前端允许的修改范围内进行加固：

1. **修复的真实前端 Bug**：
   - *文件*：`apps/web/src/app/(app)/settings/page.tsx`（用户头像与显示名称渲染处）
   - *问题根因*：在全局设置中心的团队成员管理 Tab 中，页面直接调用了 `user.name.charAt(0)` 提取头像首字符。若后端返回的某些用户记录未定义 `name` 字段，会导致前端触发 Unhandled Runtime Error，使得 Next.js 开发状态下展现红色全屏错误遮罩（或者在生产环境中直接造成白屏崩溃）。
   - *解决方案*：在此处实施了防御性代码加固，将代码重构为 `{user.name ? user.name.charAt(0) : (user.email ? user.email.charAt(0) : '?')}` 并在展示名字时使用 `{user.name || user.email || '未命名用户'}` 作为降级，消除了由于字符解析引起的运行时崩溃风险。

## 四、E2E 测试确认的 UI 行为交互

通过 Playwright 在沙箱环境中的仿真交互，我们对以下主要的 UI 功能行为进行了回归验证：
1. **Test 1**：证实系统 17 个一级路由页面可达，主区域渲染可见，未发生任何 `Application error` 或白屏，页面无残留英文标题。
2. **Test 2**：确认项目子页面文件上传入口正常、文档卡片列表可正确渲染 mock 标题、知识库检索与表单按钮状态正常。
3. **Test 3**：确认文档详情编辑器可读，点击编辑按钮可拉起 Dialog；点击“导出”能正确调用接口并无异常崩溃；侧边历史版本及评论区正常展示。
4. **Test 4**：确认设置中心的成员、角色、API密钥 Tab 顺利切换，验证了填入合法的测试邮箱可触发网络提交，且接收到了“邀请已发送”的正常 Toast/文本反馈。
5. **Test 5**：确认模板大厅卡片数据能正常反映 Mock 模板名，验证了点击“新建模板”、“更新版本”、“版本历史”按钮能拉起 Dialog。
6. **Test 6**：确认供应商通信校验（测试连接）可收到 Mock 后端的成功反馈 Toast；配额中心的刷新与报表导出交互不发生任何运行时崩溃。

## 五、后端依赖与真实环境验证建议

本轮测试的通过是基于 Playwright 本地网络拦截层实现的确定性回归。为了系统在真实生产环境中的高可靠运行，仍存在以下待确认项：

1. **需要后端能力实际支持的交互**：
   - 成员管理中实际向指定邮箱投递邀请链接与 token 的生成逻辑。
   - 知识图谱 (GraphRAG) 的批量写入与真实大语言模型分析节点、边的匹配。
   - 智能工作流设计器 (Workflow Editor) 的 DAG 配置持久化以及工作流多版本实际的引擎侧执行。
   - LLM 提供商 (openai, etc.) 与外部真实商业化接口的通信握手。

2. **真实环境验证建议**：
   - 建议在集成测试环境 (Staging) 阶段，关闭 API 拦截层，进行一次真实的非 Mock 物理回归，重点验证登录 JWT 令牌刷新、多租户数据的隔离性、以及文档真实生成的时延配额限制。

# Avenir Matrix 详细开发计划

## 1. 开发原则

本计划用于指导 Avenir Matrix 下一阶段升级开发。开发目标不是堆功能，而是把当前已有的数据模型、API 和页面组织成稳定、可交付、可审计的咨询项目文档系统。

执行原则：

- 每个迭代必须可上线、可验证。
- 先修复用户可见问题，再扩展新能力。
- 先建立项目文档主线，再做 Agent 深度编排。
- 任何 AI 自动修改下游交付物的动作都必须经过人工确认。
- 所有关键动作必须有状态、反馈、错误提示和审计记录。

## 2. 当前代码基础

### 2.1 可复用能力

后端已有基础：

- 项目、项目成员、项目邀请。
- 项目资料上传与 SourceFile。
- 文档模型、文档类型、文档状态。
- 文档版本、基线、回滚。
- 文档评论、评论解决、文档快照。
- 模板和模板版本。
- Agent Workflow、Agent Run、Agent Task。
- 知识条目、知识图谱。
- 变更请求、追溯矩阵。

前端已有基础：

- 工作台。
- 项目列表和项目详情。
- 项目资料页。
- 项目成员页。
- 项目文档页。
- 文档生成页。
- 文档详情页。
- 模板页。
- 智能体页。
- 知识图谱页。
- 可追溯性页。
- 设置页。
- 健康状态页。

### 2.2 主要缺口

- 页面和按钮交互还不稳定。
- 中英文混杂。
- 文档主线被“项目”和“文档”拆散。
- 文档版本、快照、基线、评论能力没有形成完整前端体验。
- 模板无法章节级配置。
- Skill 缺少产品化模型和管理界面。
- Agent 未按 URS/BRD/PRD 等文档类型编排。
- 发布后引用、下游影响分析、审批式同步仍未闭环。

## 3. 迭代总览

| 阶段 | 名称 | 目标 | 建议周期 |
| --- | --- | --- | --- |
| P0 | 体验稳定与可信度修复 | 消除 404、白屏、按钮无效、中英文混杂、无反馈 | 1-2 周 |
| P1 | 项目文档主线重构 | 建立以项目为中心的文档工作流 | 2-3 周 |
| P2 | 文档生命周期闭环 | 打通持续编写、版本、评论、评审、发布 | 3-4 周 |
| P3 | 模板中心升级 | 标准模板、项目模板、章节配置、提示词配置 | 3-4 周 |
| P4 | Skill 与 Agent 编排 | URS/BRD/PRD 专用 Agent 和 Skill 组合 | 4-6 周 |
| P5 | 追溯与影响分析 | 上下游引用、影响分析、审批式同步 | 4-6 周 |

## 5. P1：项目文档主线重构

### 5.1 目标

将“项目”和“文档”统一为“项目文档”主线，使用户从项目进入全部交付工作。

### 5.2 信息架构调整

左侧导航建议：

- 工作台
- 项目文档
- 知识图谱
- 智能体
- 模板中心
- 变更与追溯
- 导出
- 团队
- 系统状态
- 设置

处理方式：

- 当前 `/projects` 升级为“项目文档”主入口。
- 当前 `/documents` 改为“全局文档检索”，不作为核心一级入口，或合并进项目文档。
- 项目详情页成为项目工作台。

### 5.3 项目文档页

功能：

- 项目列表。
- 最近项目。
- 待处理文档。
- 待评审文档。
- 最近发布文档。
- 风险提示。

项目卡片动作：

- 打开项目。
- 项目设置。
- 项目成员。
- 项目资料。
- 项目文档。
- 归档项目。

验收标准：

- 项目卡片每个动作可用。
- 设置不再跳 404。
- 成员入口可邀请或添加成员。

### 5.4 项目内文档页

功能：

- 按文档类型分组。
- 按状态过滤。
- 按更新时间排序。
- 显示版本号。
- 显示评审状态。
- 显示发布状态。
- 显示待处理评论数。

验收标准：

- 每个文档只能出现在所属项目下。
- 全局文档页也必须显示所属项目。

## 6. P2：文档生命周期闭环

### 6.1 目标

让文档从生成、编辑、评论、评审、版本、发布到回滚形成完整闭环。

### 6.2 文档状态流

系统默认状态：

- 草稿
- 编写中
- 待评审
- 评审中
- 待修订
- 已批准
- 已发布
- 已归档

开发任务：

- 后端支持项目级状态配置。
- 前端显示状态流。
- 状态变更需要权限控制。
- 状态变更记录审计。

验收标准：

- 文档状态不是纯文本标签，而是可控流程。
- 非授权用户不能发布或批准。

### 6.3 持续对话式文档编写

页面结构：

- 左侧章节树。
- 中间文档正文。
- 右侧 AI 协作面板。
- 评论与版本侧栏。

功能：

- 选中章节后让 AI 补写。
- 选中文本后让 AI 改写。
- 根据评论生成修订建议。
- 根据资料补充证据。
- 保存为新版本。

验收标准：

- 不是一次生成后结束。
- 用户可以持续输入要求并更新文档。

### 6.4 版本管理

开发任务：

- 前端接入 `DocumentVersion`。
- 文档详情显示当前版本。
- 支持保存 V1、V2、V3。
- 支持查看历史版本内容。
- 支持从基线回滚。
- 支持版本说明。

验收标准：

- 点击“版本历史”不能空白或无说明。
- 没有历史版本时显示“当前仅有初始版本”。
- 回滚必须生成新的版本记录。

### 6.5 自动保存

开发任务：

- 前端编辑器每 5 分钟触发自动快照。
- 后端使用 `DocumentSnapshot` 保存。
- 页面显示自动保存时间。
- 支持从快照恢复。

验收标准：

- 自动保存失败有提示。
- 快照不污染正式版本号。

### 6.6 评论评审

开发任务：

- 支持整篇评论。
- 支持章节或段落评论。
- 支持回复评论。
- 支持标记已解决。
- 支持筛选未解决评论。

验收标准：

- 评论必须能定位到文档位置。
- 未解决评论应阻止直接发布，或给出风险提示。

## 7. P3：模板中心升级

### 7.1 目标

- 每个目录的输入字段。
- 每个目录的质量规则。

验收标准：

- 项目模板不直接修改标准模板。

- Skill 适用文档类型。
- Skill 适用行业。

Skill 类型：

- 扩散类。
- 专业类。

### 8.3 Agent 管理

建议内置 Agent：

- URS Agent。

- PRD Agent。
- User Story Agent。
- Test Case Agent。
- 需求澄清 Agent。
- 文档评审 Agent。

开发任务：

- 支持节点输入输出配置。
- 支持保存版本。

验收标准：

- 每次运行生成 AgentRun 和 AgentTask 记录。

## 9. P5：追溯与影响分析

### 9.1 目标

建立 URS、BRD、PRD、User Story、设计、测试之间的血缘追溯，并支持变更影响分析。

### 9.2 文档引用规则

规则：

- 只有已发布文档可以被下游正式引用。
- 引用必须绑定版本或基线。
- 引用关系必须记录创建者和创建时间。

建议新增模型：

- `DocumentReference`
- `DocumentImpactAnalysis`
- `DocumentSyncProposal`

### 9.3 影响分析

触发条件：

- 上游文档发布新版本。
- 上游文档内容变更。
- 上游文档被回滚。
- 需求项被新增、删除或修改。

输出：

- 受影响文档。
- 受影响章节。
- 影响等级。
- 原因说明。
- 建议动作。
- 候选修订内容。

### 9.4 审批式同步

流程：

1. 系统生成同步建议。
2. 项目负责人或文档负责人查看。
3. 选择接受、修改后接受、拒绝。
4. 接受后写入目标文档。
5. 自动生成新版本。
6. 更新追溯关系。

验收标准：

- AI 不得静默修改已发布或待评审文档。
- 所有同步必须可审计。

## 14. 风险与控制

### 14.1 产品风险

风险：功能过多导致主线不清。

控制：所有功能围绕“项目文档交付闭环”排序，不直接堆高级 Agent 能力。

### 14.2 技术风险

风险：已有模型多但语义不统一。

控制：先梳理状态、版本、快照、基线、发布的边界，再扩展功能。

### 14.3 AI 输出风险

风险：AI 自动同步下游文档造成错误传播。

控制：只生成建议，不静默应用。必须由人审批。

### 14.4 交付风险

风险：继续出现按钮不可用、页面英文、404 等低级体验问题。

控制：把 Playwright 冒烟测试纳入每次发布前置条件。
